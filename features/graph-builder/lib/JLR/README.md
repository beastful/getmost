
# JSONLang Documentation

**Version:** 1.0  
**Last Updated:** 2026-05-22

---

## 1. Overview

JSONLang is a declarative, JSON-serializable expression language designed for building web applications in both React (client) and Node.js (server) environments. Every program is a nested array — an S-expression encoded as JSON — making it trivial to store, transmit, and generate programmatically.

### Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| **JSON-only AST** | Expressions can be stored in databases, sent over WebSockets, and generated by visual editors without a parser. |
| **Async-by-default** | All arguments to native functions evaluate concurrently via `Promise.all`. I/O is composable, not sequential. |
| **Special forms** | Control flow (`if`, `then`, `lambda`) receives unevaluated arguments to enable lazy evaluation and conditional execution. |
| **Opaque constructors** | HTML tag names (`div`, `span`, `img`) are treated as data structures, not function calls. The engine preserves them as virtual DOM nodes. |
| **Gas model** | Every operation consumes gas. Runaway recursion or infinite loops are halted deterministically. |

---

## 2. Core Concepts

### 2.1 Expressions

An expression is either:

1. **A literal:** `null`, `true`, `false`, a number, or a string.
2. **An array:** `[head, ...args]` where `head` is a string determining the operation.

```json
["sum", 1, 2, 3]
```

### 2.2 Evaluation Order

Native functions evaluate **all arguments concurrently** before the function body runs:

```json
["array",
  ["fetch_json", "https://api.example.com/a"],
  ["fetch_json", "https://api.example.com/b"]
]
```
Both HTTP requests fire in parallel. The `array` function receives the results only after both resolve.

Special forms bypass this. They receive raw, unevaluated argument arrays and decide internally what to resolve and when.

### 2.3 Opaque Constructors

Any string registered as "opaque" (typically HTML tag names) is not executed. It becomes a tuple:

```json
["div", { "className": "app" }, "Hello"]
```

Evaluates to:
```javascript
["div", { className: "app" }, "Hello"]
```

The React runtime later converts this into `React.createElement`.

## 3. Built-in Special Forms

Special forms are the only constructs with **lazy evaluation semantics**. They receive a `resolve` callback to evaluate sub-expressions on demand.

### `if`

```json
["if", condition, thenBranch, elseBranch]
```

Evaluates `condition`. If truthy, evaluates `thenBranch`; otherwise `elseBranch`. The unchosen branch is never evaluated.

**Returns:** The result of the chosen branch.

### `then`

```json
["then", valueExpr, lambdaExpr]
```

Monadic bind. Evaluates `valueExpr`, then evaluates `lambdaExpr` to a function, and calls that function with the value.

**Returns:** Whatever the lambda returns.

### `seq`

```json
["seq", expr1, expr2, ..., exprN]
```

Evaluates expressions **sequentially** (not concurrently). Returns the result of the last expression.

Use this for side-effect ordering when concurrency would be non-deterministic.

### `try`

```json
["try", expr, errorHandlerLambda]
```

Evaluates `expr`. If it throws, evaluates `errorHandlerLambda` to a function and calls it with the error object.

**Returns:** Result of `expr`, or the handler's return value on error.

### `let`

```json
["let",
  ["x", 10],
  ["y", ["sum", ["get_local", "x"], 5]],
  ["get_local", "y"]
]
```

Binds variables lexically. Each binding creates a new scope. The last argument is the body.

**Returns:** Result of the body expression.

### `lambda`

```json
["lambda", ["param1", "param2"], bodyExpr]
```

Creates a first-class async function. When called, it binds parameters into a new scope and evaluates `bodyExpr`.

**Returns:** A JavaScript `Function`.

### `quote`

```json
["quote", ["anything", "here", "is", "not", "evaluated"]]
```

Returns its argument unevaluated. Useful for passing expression templates.

### `action`

```json
["action", "addTodo"]
```

Returns an opaque marker `{ __jsonlang_action: "addTodo" }`. The React runtime converts this into an event handler that looks up the action in `appDefinition.actions`.

### `run`

```json
["run", ["get", "toggleTodo"]]
```

Evaluates its argument, then if the result is a valid expression array, evaluates it again. This is the bridge between "stored expression" and "callable function".

**Critical:** Lambdas stored in `globalStore` are raw arrays. Use `run` to invoke them.

## 4. Built-in Native Functions

### 4.1 Store

| Function | Signature | Description |
|----------|-----------|-------------|
| `get` | `get(key: string) → any` | Reads from the global store. Throws if key missing. |
| `set` | `set(key: string, value: any) → any` | Writes to global store. Returns the value. |
| `del` | `del(key: string) → null` | Deletes a key from global store. |
| `get_local` | `get_local(key: string) → any` | Reads from the current lexical context. Throws if missing. |
| `set_local` | `set_local(key: string, value: any) → any` | Writes to the current lexical context. |
| `get_path` | `get_path(obj: any, ...path: string[]) → any` | Deep property access. Returns `null` if path breaks. |

**Important:** `get_local` accepts exactly **one** key. For nested access, use `get_path`:

```json
["get_path", ["get_local", "todo"], "id"]
["get_path", ["get_local", "$event"], "target", "value"]
```

| Function | Signature | Description |
|----------|-----------|-------------|
| `array` | `array(...items) → any[]` | Creates an array. |
| `concat` | `concat(...arrays) → any[]` or `concat(...strings) → string` | Concatenates arrays or strings. |

## 5. Type System

JSONLang has a lightweight, optional type registry attached to native functions. It does not enforce types at runtime by default, but the signatures are available for static validation.

### 5.1 Type Notation

Types are expressed as JSON values:

```json
"number"
"string"
"boolean"
"null"
"any"
["array", "number"]
["record", {"name": "string", "age": "number"}]
["fn", ["number", "number"], "number"]
```

### 5.2 Introspection API

```javascript
engine.getReturnType("sum");        // "number"
engine.getArgumentTypes("multiply");  // ["number"] (variadic marker)
engine.getArgumentTypes("divide");  // ["number", "number"]
```

## 6. React Runtime

### 6.1 Application Structure

```javascript
const app = {
  initialState: { count: 0 },
  actions: {
    increment: ["set", "count", ["sum", ["get", "count"], 1]]
  },
  render: ["div", {}, ["p", {}, ["get", "count"]], ["button", { onClick: ["action", "increment"] }, "+"]]
};
```

### 6.2 How Rendering Works

1. The engine evaluates `render` with a context seeded from `initialState`.
2. Opaque tags (`div`, `p`, `button`) are preserved as `[tag, props, ...children]` tuples.
3. The React runtime converts these tuples into `React.createElement` calls.
4. Event handlers (`onClick`, `onChange`) are bound at creation time. When fired, they call `engine.process()` with the handler expression and trigger a React re-render.

### 6.3 The Closure Problem

Lambda scopes created during `map` exist only during render. By the time a click handler fires, that scope is gone. **Never reference `get_local` variables inside event handlers.**

**Solution:** Stamp data onto the DOM via `data-*` attributes and read it back from `$event.target.dataset`:

```json
["span", {
  "data-id": ["get_path", ["get_local", "item"], "id"],
  onClick: ["then",
    ["to_number", ["get_path", ["get_local", "$event"], "target", "dataset", "id"]],
    ["run", ["get", "myAction"]]
  ]
}, "Click me"]
```

### 6.4 Action Binding

There are two ways to bind events:

1. **Named action:** `["action", "addTodo"]` — looks up `app.actions.addTodo`.
2. **Inline expression:** Any other array is wrapped as `{ __jsonlang_handler: true, expr: [...] }`.

## 7. Node.js Runtime

### 7.1 Application Structure

```javascript
const api = {
  port: 3000,
  routes: [
    ["GET", "/health", ["object", "status", "ok"]],
    ["POST", "/echo", ["then", ["get_local", "$body"], ["lambda", ["b"], ["get_local", "b"]]]]
  ]
};
```

### 7.2 Request Context

Each request injects:

| Variable | Type | Description |
|----------|------|-------------|
| `$request` | object | `{ method, url, headers }` |
| `$body` | any | Parsed JSON body, or raw string if unparseable |

### 7.3 Route Matching

Routes are matched by exact string equality on method and URL. No path parameters or wildcards are supported.

## 8. Security Model

### 8.1 Gas & Depth

| Limit | Default | Purpose |
|-------|---------|---------|
| `maxGas` | 50,000 | Instruction counter. Every `_eval` call increments gas. |
| `maxDepth` | 128 | Call stack depth. Prevents infinite recursion. |

When exceeded, the engine throws with path and gas usage attached.

### 8.2 Sandboxing

- No `eval` or `Function` constructor.
- No direct filesystem access in the core engine (Node extras are opt-in).
- `globalStore` is `Object.create(null)` to avoid prototype pollution.
- Only registered functions can be invoked. Unknown names throw immediately.

### 8.3 Production Hardening Checklist

- [ ] **Isolate stores per request.** The global store is shared across all concurrent requests.
- [ ] **Add request body size limits.** The server buffers all chunks without a cap.
- [ ] **Add request timeouts.** Slow `fetch_json` calls hang the event loop.
- [ ] **Sanitize errors in production.** Error responses leak internal paths.
- [ ] **Add rate limiting.** Expression evaluation is CPU-intensive.
- [ ] **Add CORS / auth middleware.** The bare `http` server has none.
- [ ] **Use `node-fetch` or `undici` explicitly.** The global `fetch` is not available in Node < 18.
- [ ] **Implement graceful shutdown.** In-flight requests are killed on `server.close()`.

## 9. Known Issues & Limitations

### 9.1 Structural

1. **Children must be spread, not wrapped.**
   ```json
   // WRONG
   ["div", {}, [["span", {}, "A"], ["span", {}, "B"]]]
   // RIGHT
   ["div", {}, ["span", {}, "A"], ["span", {}, "B"]]
   ```

2. **Plain objects are returned as-is.** If you accidentally pass an object where React expects a child, you get "Objects are not valid as a React child." This usually means a function returned an object instead of an element.

### 9.2 Semantic

3. **All lambdas are async.** You cannot write a synchronous lambda. This is usually invisible, but `reduce` and `seq` are strictly sequential because of it.

4. **`get_local` is single-key only.** Nested access requires `get_path`.

5. **Event handlers cannot close over render variables.** Use `data-*` attributes to pass data from render-time to event-time.

6. **`if` branches must be valid expressions or literals.** Because both branches are passed through `_eval`, returning a plain object from `if` is fine — but returning an unquoted array will cause the engine to try to execute it.

### 9.3 Performance

7. **Tree-walking interpreter.** Every evaluation re-resolves function names. For high-frequency operations (e.g., animation loops), consider caching compiled expressions.

8. **`map` creates `Promise.all` overhead.** Even for pure, synchronous lambdas, `map` wraps every iteration in a Promise. For large arrays (>10,000), this is expensive.

### 9.4 Node.js Server

9. **No routing parameters.** `/user/:id` is not supported; use query strings or exact paths.

10. **No middleware stack.** CORS, compression, logging, and auth must be added manually or by wrapping the server in Express/Fastify.

11. **Memory leaks in long-running processes.** The global store accumulates forever unless you manually `del` keys.

## 10. Adding User-Defined Functions

JSONLang is designed to be extended. You can add your own functions at three layers: **native functions** (eager, concurrent), **special forms** (lazy, control flow), and **opaque constructors** (preserved as data).

### 10.1 Native Functions

Native functions are the default extension point. All arguments are evaluated **concurrently** before your function body runs.

```javascript
import { JSONLangEngine } from './json-lang-engine.js';

const engine = new JSONLangEngine();

// Basic registration
engine.registerNative('double', (args) => args[0] * 2);

// With type signature (optional, for introspection)
engine.registerNative('double', (args) => args[0] * 2, {
  args: ['number'],
  ret: 'number'
});

// Variadic
engine.registerNative('max', (args) => Math.max(...args), {
  args: ['number'],
  ret: 'number',
  variadic: true
});
```

**The function signature:**

```javascript
(args, resolve, ctx, extensions) => any
```

| Parameter | Description |
|-----------|-------------|
| `args` | Array of already-evaluated arguments. |
| `resolve` | Callback to manually evaluate sub-expressions: `resolve(expr, path?)`. |
| `ctx` | The current lexical context (local variables). |
| `extensions` | The `engine.extensions` object for shared dependencies. |

**Example: a function that evaluates its second argument conditionally**

```javascript
engine.registerNative('unless_null', (args, resolve, ctx) => {
  const value = args[0];
  const fallbackExpr = args[1]; // Already evaluated! This is a problem.
  // ...
});
```

If you need **lazy evaluation**, use a special form instead.

### 10.2 Special Forms

Special forms receive raw, unevaluated argument arrays. They decide when (or if) to evaluate each argument.

```javascript
engine.registerSpecial('unless', async (args, resolve, ctx) => {
  const value = await resolve(args[0]);
  if (value === null || value === undefined) {
    return await resolve(args[1]); // Only evaluated if null
  }
  return value;
});
```

**Common use cases for special forms:**

- Conditional execution (`if`, `unless`)
- Short-circuiting (`and`, `or` — though the built-ins are eager; lazy versions would be special forms)
- Loops (`while`, `for`)
- Variable binding (`let`, `with`)
- Quoting / unquoting

**Important:** Special forms bypass the gas counter for their unevaluated arguments. The gas is only consumed when `resolve()` is called.

### 10.3 Opaque Constructors

Opaque constructors are strings that the engine treats as **data structures**, not function calls. This is how HTML tags work.

```javascript
// Register custom component names
engine.registerOpaque('chart', 'data-table', 'user-card');

// Now these are preserved as tuples:
// ["user-card", { name: "Alice" }, "Admin"]
```

The React runtime converts `user-card` into a React component lookup. You must teach your runtime how to handle custom opaques.

**Example: React runtime extension for custom components**

```javascript
// In your runtime's toReact() function:
if (type === 'user-card') {
  return React.createElement(UserCard, props, ...children);
}
```

### 10.4 Registering Extensions (Shared Dependencies)

Use `engine.extensions` to inject external libraries or configuration without polluting the global store.

```javascript
// Setup
const engine = new JSONLangEngine();
engine.extensions.db = myDatabaseClient;
engine.extensions.config = { apiKey: 'secret' };

// Inside a native function
engine.registerNative('db_find', async (args, resolve, ctx, extensions) => {
  const id = args[0];
  return await extensions.db.findById(id);
});
```

This keeps the engine testable: you can swap `extensions.db` for a mock in tests.

### 10.5 Complete Example: Adding a Validation Library

```javascript
import { JSONLangEngine } from './json-lang-engine.js';
import validator from 'validator';

const engine = new JSONLangEngine();

engine.extensions.validator = validator;

// Email validation
engine.registerNative('is_email', (args) => {
  return validator.isEmail(args[0]);
}, { args: ['string'], ret: 'boolean' });

// URL validation with options
engine.registerNative('is_url', (args, resolve, ctx, ext) => {
  const [url, options] = args;
  return ext.validator.isURL(url, options || {});
}, { args: ['string', 'object'], ret: 'boolean' });

// Sanitization (returns string)
engine.registerNative('escape_html', (args, resolve, ctx, ext) => {
  return ext.validator.escape(args[0]);
}, { args: ['string'], ret: 'string' });
```

Usage in expressions:

```json
["if", ["is_email", ["get", "userEmail"]],
  ["object", "valid", true],
  ["object", "valid", false, "error", "Invalid email"]
]
```

### 10.6 Async Functions

All native functions can be async. The engine always `await`s the result.

```javascript
engine.registerNative('db_query', async (args, resolve, ctx, ext) => {
  const sql = args[0];
  const params = args[1] || [];
  return await ext.db.query(sql, params);
}, { args: ['string', ['array', 'any']], ret: 'any' });
```

Because arguments are resolved via `Promise.all`, you can pass async sub-expressions naturally:

```json
["db_query",
  "SELECT * FROM users WHERE id = ?",
  ["array", ["get", "userId"]]
]
```

### 10.7 Functions with Side Effects

If your function mutates external state, document it clearly and consider using `seq` for ordering:

```javascript
engine.registerNative('send_email', async (args) => {
  const { to, subject, body } = args[0];
  await emailService.send({ to, subject, body });
  return { sent: true, to };
}, { args: ['object'], ret: 'object' });
```

**Warning:** Because native function arguments evaluate concurrently, side effects in sibling arguments run in an unpredictable order. Use `seq` when order matters:

```json
["seq",
  ["log", "Starting"],
  ["send_email", ["get", "emailData"]],
  ["log", "Finished"]
]
```

### 10.8 Higher-Order Functions

You can write native functions that accept and invoke lambdas:

```javascript
engine.registerNative('for_each', async (args) => {
  const arr = args[0];
  const fn = args[1];
  if (typeof fn !== 'function') throw new TypeError('for_each expects a function');
  
  for (const item of arr) {
    await fn(item); // Lambdas are async
  }
  return null;
}, { args: [['array', 'any'], 'function'], ret: 'null' });
```

Usage:

```json
["for_each",
  ["get", "users"],
  ["lambda", ["user"], ["log", ["get_local", "user"]]]
]
```

### 10.9 Best Practices

| Practice | Why |
|----------|-----|
| **Always provide type signatures** | Enables static validation and IDE-like introspection. |
| **Use `extensions` for external deps** | Keeps functions pure and testable. |
| **Document side effects** | Consumers need to know if `seq` is required. |
| **Validate argument types manually** | The signature registry is for metadata; runtime validation is your responsibility. |
| **Throw descriptive errors** | Include the function name and expected types. |
| **Avoid `eval` or `Function` in native code** | Breaks the sandbox. Use `resolve()` for sub-expression evaluation. |
| **Return consistent types** | `if` branches should return the same shape to avoid React rendering errors. |

### 10.10 Testing User-Defined Functions

```javascript
import { JSONLangEngine } from './json-lang-engine.js';

test('double function', async () => {
  const engine = new JSONLangEngine();
  engine.registerNative('double', (args) => args[0] * 2);
  
  const result = await engine.process(['double', 21]);
  expect(result).toBe(42);
});

test('db_query with mock', async () => {
  const engine = new JSONLangEngine();
  const mockDb = { query: jest.fn().mockResolvedValue([{ id: 1 }]) };
  engine.extensions.db = mockDb;
  
  engine.registerNative('db_query', async (args, resolve, ctx, ext) => {
    return await ext.db.query(args[0], args[1]);
  });
  
  const result = await engine.process([
    'db_query',
    'SELECT * FROM users',
    ['array']
  ]);
  
  expect(mockDb.query).toHaveBeenCalledWith('SELECT * FROM users', []);
  expect(result).toEqual([{ id: 1 }]);
});
```

### 10.11 Common Pitfalls

**Pitfall 1: Trying to use `get_local` for nested access**

```json
// WRONG: get_local only accepts one key
["get_local", "user", "name"]

// RIGHT: use get_path
["get_path", ["get_local", "user"], "name"]
```

**Pitfall 2: Returning an array from a native function that gets re-evaluated**

If your native function returns an expression array like `["sum", 1, 2]`, the engine will **not** re-evaluate it automatically. Use `run` or `then` to force evaluation.

**Pitfall 3: Forgetting that `map`, `filter`, `reduce` are async-aware**

Your callback will be wrapped in `Promise.all` or awaited sequentially. Do not assume synchronous execution.

**Pitfall 4: Registering a function name that collides with an opaque constructor**

If you register a native function named `"div"`, the engine will try to execute it instead of treating `<div>` as an HTML tag. Avoid naming collisions with HTML tag names.

| Fetch | `["fetch_json", "https://api.example.com/data"]` |
| Sleep | `["then", ["sleep", 1000], ["lambda", ["_"], "done"]]` |
| React element | `["div", { "className": "app" }, "Hello"]` |
| React event | `["button", { onClick: ["action", "save"] }, "Save"]` |
| DOM data pass | `["span", { "data-id": 42, onClick: handler }, "Click"]` |
| Read dataset | `["get_path", ["get_local", "$event"], "target", "dataset", "id"]` |
| Run stored lambda | `["run", ["get", "myLambda"]]` |
| User function | `["double", 21]` |
| User special form | `["unless", null, ["get", "fallback"]]` |

## 12. Glossary

| Term | Definition |
|------|------------|
| **Expression** | A JSON value that the engine can evaluate. Either a literal or an array starting with a function name. |
| **Special Form** | A construct that receives unevaluated arguments and controls evaluation order itself (e.g., `if`, `lambda`). |
| **Native Function** | A JavaScript function registered in the engine. All arguments are evaluated concurrently before it runs. |
| **Opaque Constructor** | A string (like `"div"`) that the engine treats as data, not as a function call. |
| **Gas** | An instruction counter that increments on every `_eval` call. Prevents runaway execution. |
| **Global Store** | The `engine.globalStore` object. Shared mutable state. In React it acts as the app state; in Node.js it is shared across all requests unless cloned. |
| **Context (`ctx`)** | The lexical scope chain for local variables. Created by `lambda` and `let`. |
| **Action** | A named expression stored in `appDefinition.actions`. Referenced via `["action", "name"]` in event handlers. |
| **Resolve** | The callback passed to special forms to evaluate a sub-expression on demand. |
| **Extensions** | The `engine.extensions` object for injecting external dependencies (databases, config, libraries) into native functions. |
